🔐 Cyber Security

Security Best Practices

CIA triad, OWASP Top 10, secure coding practices, and how to justify each decision for Distinction marks.

🔺 CIA Triad

The mark scheme states Distinction requires "thoroughly effective application of procedures and security controls to ensure confidentiality, integrity and availability."

🔒

Confidentiality

Ensuring data is accessible only to authorised users. Implementations: password hashing (bcrypt), HTTPS/TLS, session management, role-based access control (RBAC), data encryption at rest.

✔️

Integrity

Ensuring data cannot be tampered with or corrupted. Implementations: input validation, prepared statements, CSRF tokens, digital signatures, database transactions, checksums.

⚡

Availability

Ensuring the system is accessible when needed. Implementations: rate limiting to prevent DDoS, server-side error handling, backups, graceful degradation, error pages that don't expose system info.

🔑 Password Hashing — Explained

🚫
Never store plain-text passwords or use MD5/SHA1. These are either reversible or fast enough to brute-force. Always use a purposely slow hashing algorithm.
❌

Wrong: Plain Text / MD5

$password = "mypassword";
MD5: 34819d7beeabb9260a5c854bc85b3e44

MD5 is a fast hashing function — attackers can compute billions of MD5 hashes per second using GPU cracking. Rainbow tables can reverse common passwords instantly.

✅

Correct: bcrypt via password_hash()

bcrypt is intentionally slow (configurable cost factor). It automatically generates a random salt to prevent rainbow table attacks. password_verify() safely checks the hash without timing attacks.

PHP — Password Hashing 
<?php
// ✅ CORRECT: Hash with bcrypt (cost 12 = ~250ms on modern hardware)
$hash = password_hash($plainPassword, PASSWORD_BCRYPT, ['cost' => 12]);
// Result: "$2y$12$randomsalt+hashedvalue..." — different every time!

// ✅ CORRECT: Verify — uses constant-time comparison to prevent timing attacks
if (password_verify($plainPassword, $storedHash)) {
    // Password is correct — login the user
}

// ❌ WRONG: Never do this
if (md5($password) === $dbPassword) { // Vulnerable to rainbow tables }
if ($password === $dbPassword) { // Plain text — catastrophic breach }

📋 GDPR & Legal Compliance

📜

GDPR Key Principles

Lawfulness: Have a legal basis for processing (consent, contract).
Purpose limitation: Only use data for stated purpose.
Data minimisation: Collect only what's necessary.
Storage limitation: Don't keep data longer than needed.
Integrity & confidentiality: Keep data secure.

⚖️

Computer Misuse Act 1990

Three offences relevant to developers:
1. Unauthorised access to computer material.
2. Unauthorised access with intent to commit further offences.
3. Unauthorised modification of computer material.

As a developer, your security controls help prevent CMA violations against your users.

🏋️

Health App Specific

Health data is Special Category under GDPR (Article 9) — requires explicit consent. This includes BMI, calorie data, medical conditions. You must:
- State explicitly what health data is collected
- Allow users to delete their data (Right to Erasure)
- Provide a Privacy Notice

✅ Security Implementation Checklist

📌
Use this list to verify your Distinction-level solution covers all security aspects. The mark scheme awards Distinction for "thoroughly effective application of procedures and security controls."
  • All passwords hashed with bcrypt (cost ≥ 10)
  • All SQL queries use prepared statements
  • All output HTML-encoded with htmlspecialchars()
  • CSRF tokens in all POST forms
  • Session regenerated on login (session_regenerate_id(true))
  • Cookies set with HttpOnly, Secure, SameSite=Strict
  • Server-side authorisation on every protected route
  • Error messages don't reveal system details
  • Input validation on both client AND server side
  • GDPR Privacy Notice / consent mechanism implemented
  • Security decisions documented and justified in proposal/evaluation

❓ Common Exam Questions

Explain how prepared statements prevent SQL injection▼
In a vulnerable query like "SELECT * FROM users WHERE email = '$email'", an attacker could input ' OR '1'='1 to bypass authentication. Prepared statements separate the SQL structure from the data. The database compiles the query template first, then receives the parameters as pure data — the database engine will never interpret them as SQL commands. Using PDO: $stmt = $pdo->prepare("SELECT * FROM users WHERE email = :email"); $stmt->bindParam(':email', $email);
What is the difference between authentication and authorisation?▼
Authentication verifies who you are (login with email/password). Authorisation determines what you are allowed to do (can this user access this resource?). A logged-in user is authenticated, but may not be authorised to view another user's data. Always check both — authentication without authorisation allows horizontal privilege escalation (accessing other users' data).
Why is the Computer Misuse Act relevant to a health fitness app?▼
The Computer Misuse Act 1990 makes it illegal to access computer material without authorisation. As a developer, you must implement security controls to prevent others from committing CMA offences against your users. Specifically: implementing access controls prevents Section 1 (unauthorised access) offences; proper input validation and security controls prevent Section 3 (unauthorised modification) attacks like data tampering. Failing to implement adequate security could also expose you to civil liability under GDPR.
