🔧 Change Management

Change Management & User Support

Triggers for change, the change control process, rollback planning, traceability, and creating professional user support materials — FAQs and release notes.

⚡ Triggers for Change

Changes to software don't happen randomly — they're triggered by specific events. Understanding and documenting these triggers is part of Distinction-level iterative development evidence.

🐛

Defect / Bug

Testing reveals an error. A formal defect log entry triggers investigation, fix, and regression testing.

📋

New Requirement

Client requests additional functionality after initial scoping. Requires impact assessment before implementation.

💬

User Feedback

UAT or live user testing reveals usability problems. Feedback analysis triggers UX changes.

⚖️

Regulatory Change

New legislation (e.g. updated GDPR guidance, new accessibility standard) mandates system updates.

🔐

Security Vulnerability

A CVE (Common Vulnerability and Exposures) is published affecting a dependency. Emergency patching required.

⚙️

Technical Debt

Code that was "quick" to write but hard to maintain accumulates. Refactoring triggered to restore quality.

📈

Performance Issue

Load testing reveals the system cannot handle expected user numbers. Architecture changes triggered.

🔄

Dependency Update

A third-party library releases a new version with breaking changes or security patches.

📋 The Change Control Process

1

Change Request

Formally document: what change, why, who requested, priority

2

Impact Assessment

Analyse effect on: functionality, security, other components, timeline, cost

3

Approval

Stakeholder sign-off. May be via Change Advisory Board (CAB) in large orgs

4

Implementation

Code change in version control (new branch). Document what was changed and why

5

Testing

Unit + integration + regression testing. Ensure change doesn't break existing functionality

6

Deployment

Release to production with rollback plan ready. Monitor for issues post-deployment

7

Review & Close

Verify change met its objective. Update documentation. Close the change request.

⏪ Rollback Planning

A rollback plan defines exactly how to revert a change if it causes problems in production. Having a rollback plan is essential before deploying any significant change — it's the "undo button" for production systems.

🌿

Git Branching Strategy

Every change made on a feature branch. main always stays deployable. If a deployment fails: git revert or redeploy the previous tagged release. Never commit directly to main.

🗄️

Database Migrations

Every schema change has an "up" migration and a "down" migration. If a release breaks something, run the down migration to restore the previous schema. Tools: Flyway, Liquibase, Laravel Migrations.

📸

Snapshots & Backups

Take a full database backup before deploying. If deployment corrupts data, restore from backup. Document: where backup is stored, who can restore it, estimated restore time (RTO).

🚦

Feature Flags

New features deployed but disabled by default. Enable for a small % of users first (canary release). If problems occur, disable the flag instantly without redeploying.

Rollback Plan — Example Documentation 
ROLLBACK PLAN — Version 1.2.0 → 1.1.3
=========================================
Change: Added booking conflict detection algorithm
Date: DD/MM/YYYY | Approved by: [Name]

TRIGGER CONDITIONS (when to rollback):
  - Booking submissions fail for more than 5% of users
  - New error in error logs: "ConflictCheckException"
  - Any data loss detected in bookings table

ROLLBACK STEPS:
  1. Notify stakeholders (Slack #deployments channel)
  2. Switch load balancer to previous server instance:
       $ kubectl rollout undo deployment/resource-hub
  3. Verify v1.1.3 is serving traffic (check /health endpoint)
  4. Run DB down migration if schema changed:
       $ php artisan migrate:rollback --step=1
  5. Restore DB backup if data corruption detected:
       $ mysql resource_booking < backup_20250115_0900.sql
  6. Confirm with stakeholders that rollback is complete

ESTIMATED ROLLBACK TIME: 8 minutes
ROLLBACK OWNER: [Developer name]
POST-ROLLBACK: Open P1 incident, root cause analysis within 24hrs

👤 User Support Materials

❓

FAQs

Frequently Asked Questions anticipate common user problems and answer them proactively. Good FAQs reduce support requests and improve user experience. For Distinction: show you've considered the non-technical user perspective.

📋

Release Notes

Document what changed between versions — new features, bug fixes, known issues, breaking changes. Essential for iterative development documentation and traceability.

🎬

Screencasts / Tutorials

Video walkthroughs for complex features. Referenced in the mark scheme (Task 3A) as a feedback gathering tool — a screencast demonstrates your solution to non-technical stakeholders.

📖

User Manual

Step-by-step guide for all features. Distinction quality means separate sections for different user roles (student vs admin) and appropriate language for each audience.

Example: FAQ Document (Resource Hub)

# Resource Hub — Frequently Asked Questions (FAQ) FOR STUDENTS Q: How do I book a resource? A: Log in → Browse Resources → Click "Book Now" on your chosen item → Select date and time → Submit Booking. You'll see a "pending" status until an admin approves your request. Q: Can I book something in the past? A: No. The date picker only allows today or future dates. This is enforced both in the interface and on the server. Q: What does "pending" mean for my booking? A: Your booking has been submitted and is waiting for admin approval. You'll see it update to "approved" or "rejected" after review. You can cancel a pending booking from "My Bookings". Q: I tried to book but got "resource already booked"? A: Another student has an approved or pending booking for that time slot. Try a different time or a different resource of the same type. FOR ADMINISTRATORS Q: How do I approve or reject a booking? A: Admin Dashboard → Manage Bookings → Find the pending booking → Click Approve or Reject. The student's dashboard updates immediately. Q: How do I mark a resource as under maintenance? A: Admin Dashboard → Resources → find the resource → Set Maintenance. It will disappear from student booking options until set back to Available.
