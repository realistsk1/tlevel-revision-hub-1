WCAG 2.1 & Accessibility

The four POUR principles, key WCAG success criteria, ARIA roles, and how to write Distinction-level accessibility justifications.

🌊 The POUR Principles

👁️

Perceivable

Information and UI components must be presentable to users in ways they can perceive. It can't be invisible to all their senses.

Examples: alt text, captions, sufficient colour contrast, don't use colour alone

⌨️

Operable

UI components and navigation must be operable. Users shouldn't need to perform interactions they can't perform.

Examples: keyboard navigation, no seizure-inducing content, skip navigation, sufficient time

🧠

Understandable

Information and the operation of the UI must be understandable. Users shouldn't encounter content or operations beyond their understanding.

Examples: language attribute, error identification, labels and instructions, consistent navigation

🔧

Robust

Content must be robust enough that it can be interpreted by a wide variety of user agents, including assistive technologies.

Examples: valid HTML, name/role/value for all components, status messages programmatically determinable

📋 Key WCAG 2.1 AA Success Criteria

CriterionLevelWhat You Must DoCode Solution
1.1.1 Non-text Content A All images have text alternatives alt="descriptive text" or alt="" for decorative
1.4.3 Contrast (Minimum) AA 4.5:1 for normal text; 3:1 for large text (18pt+) Verify with WebAIM Contrast Checker
1.4.4 Resize Text AA Text can be resized up to 200% without loss of content Use relative units: rem, em, % — never px for font sizes
2.1.1 Keyboard A All functionality available via keyboard Tab order, visible focus, no keyboard traps
2.4.3 Focus Order A Focusable components receive focus in a logical order Logical DOM order; use tabindex="0" for custom components
2.4.6 Headings and Labels AA Headings and labels describe topic or purpose Use h1–h6 in logical hierarchy; use <label for="...">
3.1.1 Language of Page A Default human language can be programmatically determined <html lang="en">
3.3.1 Error Identification A Input errors automatically detected and described to user role="alert" + aria-live="polite" + descriptive message
4.1.2 Name, Role, Value A All UI components have a name, role, and value Semantic HTML; ARIA where needed (aria-label, role)

💻 ARIA Code Examples

HTML — ARIA Best Practices 
<!-- ✅ Skip navigation link (keyboard users skip to content) -->
<a href="#main-content" class="skip-link">Skip to main content</a>

<!-- ✅ Semantic landmark regions -->
<header role="banner">...</header>
<nav aria-label="Main navigation">...</nav>
<main id="main-content">...</main>
<footer role="contentinfo">...</footer>

<!-- ✅ Icon button with accessible name -->
<button aria-label="Close dialog">
  <svg aria-hidden="true">...</svg>
</button>

<!-- ✅ Dynamic content update announcement -->
<div
  id="workout-status"
  role="status"
  aria-live="polite"
  aria-atomic="true"
></div>

<!-- ✅ Expandable accordion —  ARIA states -->
<button
  aria-expanded="false"
  aria-controls="section-content"
>
  Section Title
</button>
<div id="section-content" hidden>
  Content here
</div>

<!-- ✅ Form error linking -->
<label for="weight">Weight (kg)</label>
<input
  type="number"
  id="weight"
  aria-required="true"
  aria-describedby="weight-error"
  aria-invalid="false"
/>
<span id="weight-error" role="alert"></span>

❌ Common Accessibility Mistakes

Using outline: none on focused elements▼
Removing focus outlines makes keyboard navigation impossible for sighted users. Never just remove it. Instead: :focus-visible { outline: 3px solid #3d5af1; outline-offset: 3px; } — this shows focus for keyboard users but not mouse clicks.
Using colour alone to indicate errors▼
If an error field only turns red, 8% of men (colour-blind users) won't perceive it. Always combine: colour change + icon + text description. WCAG 1.4.1 "Use of Colour" prohibits colour as the sole visual means of conveying information.
Missing or generic alt text▼
Bad: alt="image" or alt="photo123.jpg"
Good: alt="Bar chart showing weekly calories burned, with Sunday as the highest day at 450 calories"
Decorative: alt="" — screen readers skip it entirely, which is correct.
Using divs as buttons▼
<div onclick="..."> has no role, no keyboard activation, no focus behaviour. Screen readers can't announce it as a button. Use <button> which has all of these by default. If you must use a div: add role="button", tabindex="0", and handle both click and keypress events.

⚖️ Legal Basis — Equality Act 2010

⚠️
The Equality Act 2010 requires service providers (including websites) to make reasonable adjustments for people with disabilities. Knowingly building an inaccessible web application could constitute disability discrimination. WCAG 2.1 AA is the de facto standard used by courts and regulators to determine compliance.
