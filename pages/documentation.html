🔄 Iterative Development Lifecycle

💡
The mark scheme requires evidence of an iterative development process with four phases: Requirements → Design → Implementation & Test → Review. Distinction requires "thorough and detailed recording of notable changes" and "convincing and perceptive rationales."
📋

1. Requirements

Gather and analyse requirements. Produce completed functional/non-functional requirements spec. Document user stories. Create initial risk assessment.

✏️

2. Design

Design the solution: wireframes, ERD, algorithm pseudocode/flowcharts, test strategy. Design documentation must be audience-appropriate (technical and non-technical).

⚙️

3. Implement & Test

Code, integrate, and test. Apply test strategy. Log all defects. Execute regression testing after each fix. Version control every change.

🔍

4. Review

Evaluate software against requirements. Gather user feedback. Propose changes. Decide: discard this iteration or use as basis for next. Document rationale for all decisions.

📖 User Stories

Format: "As a [type of user], I want to [goal] so that [reason]." Each story should have acceptance criteria.

US-06 As a site administrator, I want to view all users and their membership levels so that I can manage subscriptions. Admin role required; list paginated; search by email; cannot access user health data without consent Could Have

⚠️ Risk Assessment

For Distinction: identify risks across security, data protection, physical harm context, and development risks. Each should have a likelihood, impact, and concrete mitigation.

Risk IDRiskCategoryLikelihoodImpactMitigation
R-01 SQL injection attack exposes user health data Security High Critical PDO prepared statements on all queries; input validation; principle of least privilege DB user
R-02 Data breach exposing password hashes Security Medium High bcrypt hashing (cost 12); no plain-text storage; GDPR breach notification procedure within 72 hours
R-03 Incorrect calorie/BMI calculations cause users to follow harmful dietary advice Physical Harm Medium High Unit-tested calculations against WHO reference values; prominent disclaimer "consult GP before starting plan"; out-of-range value warnings
R-04 User selects training intensity beyond their capability causing injury Physical Harm Medium Medium Fitness level assessment before plan generation; beginner/intermediate/advanced classification; GP consultation disclaimer; T&Cs requiring user responsibility
R-05 GDPR non-compliance (processing special category health data without consent) Legal High if unchecked Critical (fines up to 4% global turnover) Explicit consent checkbox at registration; Privacy Notice; consent withdrawal mechanism; Data Protection Officer designated
R-06 Third-party API downtime breaks workout video feature Technical Medium Low Graceful degradation — show text descriptions if API unavailable; cache previously loaded videos; user notification

🖼️ Wireframe Annotation Guide

📐

What to Annotate

Good wireframes annotate: element purpose, interaction behaviour, responsive behaviour, accessibility notes (e.g. "tab order: 1→2→3"), content strategy (placeholder text vs real content).

🎨

Design Heuristics (Mark Scheme)

The mark scheme references Nielsen's heuristics. For Distinction: minimise memory load, use consistent conventions, provide appropriate feedback, aesthetic and minimalist design, good use of space and visual hierarchy.

📱

Responsive Considerations

Show at least two breakpoints. Mobile-first approach: design for smallest screen first, then add complexity for larger screens. Annotate which elements collapse/stack on mobile.

⚠️
Mark scheme note: "The design should not contain information which is irrelevant or rarely needed. Every piece of extra information competes with the relevant information and diminishes their relative visibility." This directly references Nielsen's Aesthetic & Minimalist Design heuristic — your UI annotations should justify why elements are included, not just describe them.
