🔒 Rate Limiting — Distinction Level

index.php — Lines 15–36 
const MAX_LOGIN_ATTEMPTS = 5;
const LOCKOUT_SECONDS    = 900; // 15 minutes

function isLockedOut(): bool
{
    if (!isset($_SESSION['login_attempts'])) return false;
    if ($_SESSION['login_attempts'] >= MAX_LOGIN_ATTEMPTS) {
        return (time() - ($_SESSION['last_attempt_time'] ?? 0)) < LOCKOUT_SECONDS;
    }
    return false;
}

function recordFailedAttempt(): void
{
    $_SESSION['login_attempts']    = ($_SESSION['login_attempts'] ?? 0) + 1;
    $_SESSION['last_attempt_time'] = time();
}

function resetAttempts(): void
{
    unset($_SESSION['login_attempts'], $_SESSION['last_attempt_time']);
}
⭐
Distinction — Rate Limiting (Availability / CIA Triad)

This implements brute-force protection — 5 attempts then 15-minute lockout. The mark scheme awards Distinction for "thoroughly effective application of procedures and security controls to ensure confidentiality, integrity and availability." Rate limiting directly protects availability (prevents DoS-style brute force) and confidentiality (stops password guessing). Use of PHP constants for magic numbers (MAX_LOGIN_ATTEMPTS) also earns code organisation marks.

✅

'Invalid username or password.' is intentionally ambiguous — this is the correct security practice. Specific messages like "that username doesn't exist" enable user enumeration attacks. Mention this explicitly in your evaluation: "I used a generic error message to prevent user enumeration."

Improvement — Add PHPDoc comment to each function

For Distinction code organisation marks you need "informative commenting." Add: /** Checks if the current session has exceeded the maximum login attempts. @return bool True if locked out. */ above each function. Currently zero docblock comments exist in this file.

auth.php — CSRF verification with constant-time comparison 
function verifyCsrf(): bool
{
    return isset($_POST['csrf_token'], $_SESSION['csrf_token'])
        && hash_equals($_SESSION['csrf_token'], $_POST['csrf_token']);
}
⭐
Distinction — hash_equals() prevents timing attacks

Using === for string comparison leaks information through execution time — an attacker can measure microsecond differences to determine how many characters matched. hash_equals() uses constant-time comparison regardless of where strings differ. This is an advanced security detail that goes beyond what most T-Level students implement. Mention this explicitly in your evaluation — it demonstrates deep security knowledge that the mark scheme calls "perceptive."
