Mock OSP Practice Tasks

Practice tasks matching the format of the real exam. Each task includes the brief, mark allocation, and Distinction-level guidance.

📌
The OSP (Occupational Specialism Project) is a real project you build over time, not a written exam. These mock tasks help you practise the type of evidence you need to produce. Focus on producing documentation and code that matches Band 3 (Distinction) descriptors.

Task 1 Activity A — The Proposal

Assessment criteria: Decomposing the problem (9) · Wider Issues (9) · Business Context (6)

24 marks

📋 Brief

A local gym chain, FitLife, wants to develop a digital health and fitness platform. The platform should allow members to track workouts, calculate health metrics (BMI, BMR), receive personalised plans, and access video content. It must have free and premium membership tiers. You have been commissioned to develop this platform.

Produce a proposal that includes:

📍 Part 1: Decompose the Problem (9 marks)▼

What to produce: Break down the complete problem into sub-problems. Identify all functionality the system needs. Decompose each area into inputs, processes, and outputs.


Distinction guidance: "Fully identifies the problems to be solved; effectively decomposes the problems identified; proposed solution would meet the full needs of the client and users; mitigate the potential risks; address relevant regulatory guidelines and legal requirements."


Suggested structure:

  • User Management: registration, login, profile, membership tiers
  • Workout Tracking: log entry, history, analytics, data visualisation
  • Health Calculations: BMI, BMR, calorie tracking (inputs, formula, output, edge cases)
  • Personalised Plans: algorithm logic, free vs premium gate
  • Content: API integration for video content, social features
  • Administration: user management, reporting
🌐 Part 2: Wider Issues & Rationale (9 marks)▼

Distinction guidance: "Comprehensive lines of reasoning that fully justify how: the recommended solution meets client needs; potential risks will be mitigated; the proposed solution will address relevant regulatory guidelines and legal requirements."


You must cover:

  • Security risks: SQL injection, XSS, CSRF, broken authentication — each with specific technical mitigations
  • Privacy: GDPR, special category health data, consent, right to erasure
  • Physical harm: Incorrect workout instructions, inappropriate fitness levels — disclaimers, GP referral, age verification
  • Legal: Computer Misuse Act, Equality Act (accessibility), Consumer Rights Act
  • Emerging tech context: Wearable device integration, mobile-first, cloud hosting, REST APIs
🏢 Part 3: Business Context (6 marks)▼

Distinction guidance: "Comprehensive and perceptive definitions of: functional and non-functional requirements; key performance indicators; user acceptance criteria."


Produce all three with measurable, specific criteria. Avoid vague statements like "the app should be fast" — instead write "95th percentile page load ≤ 2.5s on 4G connection (Lighthouse ≥ 90)."


See the Distinction Examples page for a full Distinction-level answer to this section.

Task 1 Activity B — The Design

Interface Design (6) · Algorithm Design (10) · Data Requirements (6) · Test Strategy (6) · Communication (6)

34 marks
🎨 Interface Design (6 marks)▼

Produce wireframes for at least 3 key screens: Dashboard, Workout Log, BMI Calculator. For Distinction: "sophisticated and highly effective use of layout and white space, visual hierarchies, and common conventions."


Your wireframes must show: Navigation structure, content hierarchy, form elements with labels, responsive layout indicators, accessibility notes (tab order, ARIA regions), feedback message locations, error state designs.


Annotate with HCI principles: Nielsen's heuristics, Fitts's Law (button sizing), progressive disclosure (don't show all options at once).

🧮 Algorithm Design (10 marks)▼

Design algorithms for at least 3 key processes using flowcharts, pseudocode, or data flow diagrams. For Distinction: "precise logic; efficient structure and sequence; effective and consistent use of accepted conventions."


Suggested algorithms to design:

  • User Authentication: Input email/password → validate format → query DB → verify hash → regenerate session → redirect
  • BMI Calculation: Input height/weight → validate range → calculate BMI = weight / (height²) → classify → return result + advice
  • Personalised Plan Generator: Input fitness level + goals → check membership tier → filter workout database → apply progressive overload algorithm → output 7-day plan
  • Calorie Tracking: Input exercise type + duration → lookup MET value → calculate calories = MET × weight × hours → update daily total → check against target
🗄️ Data Requirements (6 marks)▼

Produce an ERD with at minimum: users, workouts, workout_types, memberships tables. For Distinction: "fully appropriate variables, data structures, data types; thoroughly appropriate and consistent naming conventions; thoroughly effective error handling procedures."


Include a data dictionary: for each attribute list the name, data type, size, constraints, and example value. Demonstrate normalisation (3NF) — no repeating groups, all non-key attributes depend only on the primary key.

🧪 Test Strategy (6 marks)▼

Design your test approach before you build. For Distinction: "thorough and detailed understanding of how components interrelate, the order in which components should be tested, the types of test that are required."


Use a phased approach: Unit testing → Integration testing → System testing → UAT. Specify which types of test data (normal, erroneous, boundary) are needed for each component. Plan for regression testing. Reference black-box vs white-box testing explicitly.

Task 2 — Developing the Solution

Functionality (8) · Code Organisation (8) · User Experience (8) · Legal/Regulatory (6) · Testing (12) · Documentation (6)

48 marks
⚠️
Must use two different languages (e.g. PHP + SQL, Python + SQL). Band 2+ requires at least two languages. Band 4 (Distinction) requires "consistently efficient functional code in at least two different languages."
⚡

Functionality Checklist

✅ Works in 2+ languages
✅ No major errors persist
✅ Efficient code (no unnecessary loops)
✅ Precise logic throughout
✅ Modular structure (functions/classes)
✅ Server-side scripting with request/response

📁

Code Organisation Checklist

✅ Consistent naming conventions (camelCase JS, snake_case PHP)
✅ Logical folder structure (MVC)
✅ Informative comments on all non-obvious logic
✅ No copy-pasted procedural code
✅ Constants not magic numbers
✅ PHPDoc / JSDoc on all functions

😊

User Experience Checklist

✅ Useful — outputs accurate
✅ Usable — intuitive, clear navigation
✅ Desirable — consistent visual design
✅ Accessible — WCAG 2.1 AA
✅ Credible — real, validated health data
✅ Fully robust error handling

Task 3 — Gathering & Evaluating Feedback

Feedback Materials (18) · Evaluation of Outcomes (9) · Content & Assets (6)

33 marks
📊 Part A: Gathering Feedback (18 marks)▼

Produce materials to gather feedback from different user types. For Distinction: "materials would allow for the gathering of high-quality feedback; consistently provides the opportunity for evidence-informed further iteration."


Include:

  • A screencast/demo video annotated for non-technical stakeholders
  • A structured questionnaire for end users (gym members) with Likert scale and open questions targeting specific FRs and UX dimensions
  • A technical code review checklist for programming professionals covering security, performance, maintainability
  • An observation log template for watching test users complete key tasks (record: task completion rate, errors made, time on task)

Cover different audiences: gym users of varied ages/abilities, gym managers, and a programming professional peer reviewer.

📈 Part B: Evaluating Feedback (15 marks)▼

For Distinction: "thorough and detailed evaluation of how well the prototype meets: functional and non-functional requirements; KPIs; user acceptance criteria. Convincing and perceptive rationale for future iteration."


Structure your evaluation:

  • For each FR: state whether it was met, with evidence from testing/feedback
  • For each NFR: provide measured data (e.g. Lighthouse score, Axe test results)
  • For each KPI: explain what you measured and what data you collected
  • For each UAC: state pass/fail with evidence
  • For future iterations: be specific — don't say "I would improve the UI", say "Based on user observation data showing 40% task failure rate on the BMI calculator, iteration 2 would restructure the input form to use a step-by-step wizard interface, reducing cognitive load."
