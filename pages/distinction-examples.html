⭐ Distinction Examples

Pass vs Merit vs Distinction

Side-by-side answer comparisons based directly on the official mark scheme criteria. Learn what "comprehensive lines of reasoning" looks like in practice.

🎯
These examples are based on the T-Level Digital Production, Design & Development Sample Assessment Material Mark Scheme. Distinction = Band 3 language: "comprehensive, thorough, perceptive, convincing, fully, consistently."

🔐 Security Justification (Task 1A — Wider Issues)

Mark Scheme Focus: "Appreciation of Wider Issues in Context" — up to 9 marks. Distinction requires "comprehensive lines of reasoning that fully justify how potential risks will be mitigated."

🟡 Pass — Band 1 (1–3 marks)

My solution will be secure. I will use passwords to protect user accounts. The app will also be safe from hackers. I will store user data in a database and make sure only logged-in users can see it. There are laws about data like GDPR that I will follow.


Why this is Pass: Identifies some risks (hacking, data storage) but provides limited reasoning. No specific technical implementations. Mentions GDPR without explaining what it requires. No mitigation strategies — just vague reassurances.

🔵 Merit — Band 2 (4–6 marks)

My health and fitness app will handle sensitive personal data including health metrics such as BMI and calorie intake. This data is considered special category under GDPR, which requires explicit user consent before processing. I will implement a consent mechanism during registration.

To protect user accounts, I will store passwords using the PHP password_hash() function with bcrypt, which prevents attackers from recovering plain-text passwords if the database is breached. SQL injection attacks will be prevented using PDO prepared statements.

The Computer Misuse Act 1990 makes unauthorised access illegal — my authorisation system will ensure users can only access their own data.


Why this is Merit: Good reasoning for most risks. Mentions specific technologies. However, doesn't comprehensively cover all risks (no mention of XSS, CSRF, availability, session management). Doesn't fully explain why each technology solves the specific problem.

🟣 Distinction — Band 3 (7–9 marks)

My health and fitness app processes Special Category personal data under GDPR Article 9 (health and biometric data including BMI, calorie intake, and workout intensity). This imposes strict legal obligations: I must obtain explicit, informed consent at registration, implement a clear Privacy Notice explaining what data is collected and why, and provide mechanisms for users to exercise their rights including the Right to Erasure (Article 17) and Data Portability (Article 20). All personal data will be minimised — only data strictly necessary for the platform's functionality will be collected.

The three primary technical security risks and their mitigations are:

  • SQL Injection: Without mitigation, an attacker could manipulate database queries via form inputs to extract all user health data or bypass authentication entirely. I will use PDO prepared statements throughout — these separate SQL structure from user data at the database engine level, making it impossible for input to be interpreted as SQL commands. This directly preserves data integrity (CIA triad).
  • Broken Authentication: Plain-text or weakly hashed passwords would expose all users if the database were breached. I will use PHP's password_hash() with bcrypt at cost factor 12, which makes each hash take ~250ms — computationally infeasible to brute-force. Salts are generated automatically per-hash, preventing rainbow table attacks. Sessions will be regenerated on login using session_regenerate_id(true) to prevent session fixation. This preserves confidentiality.
  • XSS & CSRF: Cross-site scripting attacks could allow attackers to steal session cookies via injected JavaScript. All output will be sanitised using htmlspecialchars() and a Content-Security-Policy header will restrict script sources. CSRF attacks, which could trick authenticated users into submitting malicious requests, will be mitigated using synchronised token pattern — a per-session cryptographic token in all state-changing forms. This preserves integrity and protects user availability.

Under the Computer Misuse Act 1990, my security controls are essential to prevent Section 1 (unauthorised access) offences against users. Inadequate security would leave me civilly liable under GDPR for failing to implement "appropriate technical measures" (Article 32). Additionally, under the Health and Safety at Work Act, any incorrect workout instructions generated by my algorithms that cause physical injury represent a contextual risk — I will implement clear disclaimers and recommend users consult a GP before following personalised plans.


Why this is Distinction: Comprehensive identification of all risk categories (data law, authentication, injection, XSS/CSRF, physical harm). Each risk has a specific, technically accurate mitigation with the reasoning chain explained. Named legislation applied to specific scenarios. CIA triad explicitly addressed. Covers the full business/industry context.

🧪 Testing Strategy (Task 1B / Task 2)

Mark Scheme Focus: "Test Strategy" up to 6 marks + "Suitability of Test Data" up to 6 marks. Distinction: "thorough and detailed understanding of how components interrelate, the order in which components should be tested."

🟡 Pass (1–2 marks)

I will test my app by testing each part to make sure it works. I will try logging in with correct and incorrect passwords. I will also test the workout tracking feature. If something breaks I will fix it and test again.

Why Pass: Some understanding of testing exists but no test types named, no data types described, no order or methodology. No link between tests and requirements.

🔵 Merit (3–4 marks)

I will use a combination of white-box and black-box testing. White-box testing will check the internal logic of my algorithms — for example, the BMI calculation function will be unit tested with known inputs to verify correct outputs. Black-box testing will test the user-facing features such as the login form and workout submission.

Test data will include normal data (valid email and password), erroneous data (invalid email format, SQL injection strings), and boundary data (password exactly 8 characters, 7 characters). I will use regression testing after each fix to ensure new changes don't break existing functionality.

Why Merit: Good understanding, correct test types named, test data categories mentioned. Doesn't fully explain the order of testing, integration between components, or trace tests back to specific functional requirements.

🟣 Distinction (5–6 marks)

My test strategy follows a bottom-up integration testing approach, reflecting the component dependency chain of my MVC architecture:

  • Phase 1 — Unit Testing (White-box): Test individual functions in isolation. The Database class will be tested first as all Models depend on it. Each Model method (e.g. Workout::getByUserId()) will be unit tested with known PDO mock data to verify correct SQL and data mapping. The BMI and calorie calculation algorithms will be unit tested against medically accepted reference values, including boundary values (underweight/normal weight boundary: BMI = 18.5) and extreme values (BMI = 0, negative inputs).
  • Phase 2 — Integration Testing: Once units pass, test component interactions. Controller ↔ Model integration: verify that WorkoutController::store() correctly calls Workout::create() and returns the expected response. Front-end ↔ Back-end: verify AJAX fetch requests to the API correctly receive and render JSON workout data.
  • Phase 3 — System Testing (Black-box): Full end-to-end tests against the complete application. Each functional requirement (FR) will have a corresponding test case: FR-01 "User can register" → test with valid data, duplicate email, missing fields, invalid email format. Test data types: normal (valid email/password), erroneous (SQL injection strings: ' OR '1'='1, XSS payloads: <script>alert(1)</script>), boundary (password = 7 chars, 8 chars, 255 chars).
  • Phase 4 — User Acceptance Testing (UAT): Distribute a screened prototype to representative testers (gym users, fitness professionals) with structured questionnaires measuring usability, accuracy of health data, and performance on mobile devices.

Regression testing will be applied after every defect fix, re-executing the full Phase 1 and Phase 2 test suites to ensure no new defects have been introduced. Defects will be tracked in a test log with ID, description, severity, fix applied, and retest result.

Why Distinction: Thorough, structured, phase-based approach. Explains why components are tested in that order (dependency chain). Covers all three test data types with specific examples. UAT links to real users. Regression testing explicitly planned. Tests traceable to functional requirements.

♿ Accessibility Reasoning (Task 2 — Legal & Regulatory)

Mark Scheme Focus: "Legal and Regulatory Guidelines and Standards" — Distinction: "consistent and effective application of standards and guidelines in relation to accessibility, compatibility, legal and ethical considerations."

🟡 Pass

I made my website accessible by using alt text on images and making the text big enough to read. I also used high contrast colours so people can see it easily.

🔵 Merit

My application follows WCAG 2.1 Level AA guidelines. I implemented alt attributes on all meaningful images (alt="" for decorative images so screen readers skip them). All form inputs have associated <label> elements to support screen reader navigation. Colour contrast ratios meet the minimum 4.5:1 requirement for normal text. The application is navigable by keyboard alone — all interactive elements receive focus in a logical tab order.

🟣 Distinction

My health and fitness application implements WCAG 2.1 Level AA compliance across all four principles (POUR):

  • Perceivable: All images carry descriptive alt attributes (decorative images use alt=""). Colour is never the sole means of conveying information — for example, error states use both red text AND an error icon AND an ARIA role="alert" message. Contrast ratios for body text ≥ 4.5:1 and for large text ≥ 3:1, verified using the WebAIM Contrast Checker. Video content would include captions.
  • Operable: All functionality is accessible via keyboard. Custom interactive components (accordion menus, modal dialogs) implement the ARIA authoring practices keyboard patterns (e.g. Escape closes modals, arrow keys navigate menu items). No content flashes more than 3 times per second (seizure risk). Focus indicators are clearly visible — I removed outline: none and replaced it with a 3px contrasting ring.
  • Understandable: The <html lang="en"> attribute enables correct screen reader pronunciation. Error messages are specific and suggest how to fix the problem (not just "invalid input"). All form fields use autocomplete attributes to support users with cognitive disabilities.
  • Robust: Semantic HTML5 elements (<nav>, <main>, <article>) provide structural meaning to assistive technologies without relying on ARIA. ARIA roles and attributes are used only where native semantics are insufficient. The application is tested with NVDA screen reader and Chrome Axe DevTools.

This is a legal requirement under the Equality Act 2010 which prohibits discrimination based on disability. As a health platform, accessibility is particularly important — users with visual impairments, motor disabilities, or cognitive conditions must be able to access fitness and health information equitably.

🏗️ System Design Decisions (Task 1A — Business Context)

Mark Scheme Focus: "Appreciation of Business Context" — Distinction: "comprehensive and perceptive definitions of functional and non-functional requirements, key performance indicators, user acceptance criteria."

🟡 Pass

Functional requirements: Users can log in, track workouts, view their progress.
Non-functional requirements: The app should be fast and easy to use.
KPIs: The app should work well.
User acceptance: Users should be happy with the app.

🔵 Merit

Functional Requirements (what the system does):

  • FR-01: Users can register with email and password
  • FR-02: Users can log in and out securely
  • FR-03: Users can log workouts with type, duration, and calories
  • FR-04: Users can view their workout history
  • FR-05: Premium members can access personalised workout plans

Non-functional Requirements (how well):

  • NFR-01: Page load time under 3 seconds
  • NFR-02: 99% uptime
  • NFR-03: WCAG 2.1 AA compliant

KPIs: Number of daily active users, average session duration, workout log completion rate

🟣 Distinction

Functional Requirements (what the system must do — verifiable by testing):

  • FR-01: Users shall register using a unique email address and password meeting complexity requirements (min 8 chars, 1 number, 1 symbol)
  • FR-02: The system shall authenticate users and issue a session token; sessions expire after 30 minutes of inactivity
  • FR-03: Authenticated users shall log a workout entry with: activity type, duration (minutes), estimated calories, date/time
  • FR-04: The system shall calculate and display BMI given user's height and weight, classified per WHO guidelines
  • FR-05: Premium members shall access AI-generated personalised weekly workout plans; free members see a paywall with upgrade prompt
  • FR-06: The system shall integrate with a REST API to fetch exercise video content for the specified workout type

Non-Functional Requirements (quality attributes — measurable):

  • NFR-01 Performance: 95th percentile page load ≤ 2.5s on a 4G connection (Lighthouse score ≥ 90)
  • NFR-02 Availability: 99.5% uptime (≤ 3.65 hours downtime/year)
  • NFR-03 Security: OWASP Top 10 compliance; GDPR Article 32 compliant encryption
  • NFR-04 Accessibility: WCAG 2.1 Level AA — verified by automated (Axe) and manual screen reader testing
  • NFR-05 Scalability: Architecture must support horizontal scaling to 10,000 concurrent users
  • NFR-06 Maintainability: Code must score ≥ B on CodeClimate; all functions documented with PHPDoc

Key Performance Indicators (business success metrics):

  • KPI-01: Free-to-premium conversion rate ≥ 5% within first month
  • KPI-02: Daily Active Users (DAU) / Monthly Active Users (MAU) ratio ≥ 0.2 (stickiness)
  • KPI-03: Workout log completion rate ≥ 60% of sessions started
  • KPI-04: User satisfaction score (NPS) ≥ 40 from post-launch survey

User Acceptance Criteria (conditions for client sign-off):

  • UAC-01: A test user can complete registration → login → log a workout → view history in ≤ 3 minutes without guidance
  • UAC-02: All WCAG 2.1 AA automated test violations = 0 on production build
  • UAC-03: All functional requirements verified by independent QA tester with no critical defects outstanding
  • UAC-04: Load test confirms ≥ 500 concurrent users without performance degradation
